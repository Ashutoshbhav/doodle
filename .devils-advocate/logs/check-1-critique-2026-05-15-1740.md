## High-Risk Area Findings

### 1. CAPTURE INTEGRITY — REAL RISK (the headline finding)
waitlist.ts:39-50 and 57-68 both return `{ok:true}` when the row is NOT written:
- **No DATABASE_URL** → `console.warn` + success. On Vercel, `console.warn` goes to ephemeral function logs that age out. If `DATABASE_URL` env var is missing/typo'd in the Vercel project, **100% of paid signups are lost** and the user, the Pixel, and Google Ads all report success. The conversion KPI (signup count in the DB) reads zero while ad platforms report leads — silent, total miscount.
- **Insert throws** → Sentry capture + success. Sentry `extra:{email}` preserves the lead IF the Sentry DSN is configured and quota not exhausted. There is no DB reconciliation path; recovering leads means scraping Sentry issues by hand.
- **`onConflictDoNothing`** → a repeat email returns `{ok:true}`, fires `fbq('Lead')` again. Pixel over-counts repeat submitters with no `eventID` dedupe.
- **normalizeSource attribution — CORRECT.** Input `lcl_may26:meta:m1` → trim/lower → not in ONSITE set → strip to `[a-z0-9:_-]` keeps `:` → `lcl_may26:meta:m1` preserved intact (≤60). Per-creative attribution survives. `token()` on page already strips `:` from each UTM part so injection of extra colons isn't possible. **This part is sound.**

**Fix:** For a paid campaign the conversion metric MUST equal stored rows. (a) Add a startup assertion / health check that `getDb()` is non-null in production (fail the deploy or alert if `DATABASE_URL` unset). (b) On insert failure, still show success to the user (good UX) BUT do not fire the conversion pixel, and write the lead to a durable fallback (append to a Vercel KV / Blob / second table) — Sentry is an error tracker, not a lead store. (c) Distinguish insert vs conflict (use `.returning()` and check rowcount) so Pixel `Lead` fires only on a genuinely new row.
